Фаза 1: Фундамент, Ингестия и Синхронизация (Спринты 1–2)
Цель: Создать инфраструктуру, загрузить исторические данные (Seeding), настроить их ежедневное обновление (Sync) и подготовить чистую базу аннотаций без использования полного текста PDF.
Спринт 1: Ядро и Пайплайн NASA (Seeding & Sync)
Задача 1.1: Настройка окружения. Развертывание RavenDB 7.0 (Docker) и создание ASP.NET Core Web API (Clean Architecture).
Задача 1.2: Проектирование моделей. Создание C# DTO. Строгий учет Nullable<double> для масс и радиусов.
Задача 1.3: Fluent ADQL Builder. Создание кастомного билдера (IAdqlQueryBuilder) для безопасной генерации SQL-подобных запросов без "магических строк".
Задача 1.4: Базовый клиент NASA (TAP). Настройка HttpClient с FORMAT=json и кастомными JsonConverter (для обхода бага с числами .85). Интеграция Polly (Retry with Exponential Backoff).
Задача 1.5: Стратегия Seeding (Первичная загрузка). Скрипт SeedFullCatalogAsync() для выкачивания всех ~6200+ планет из таблицы pscomppars с использованием пагинации (OFFSET/TOP).
Задача 1.6: Инкрементальный Sync. Настройка фоновой джобы (например, Quartz.NET), которая раз в сутки забирает только изменившиеся данные (rowupdate > last_sync).
Спринт 2: arXiv OAI-PMH и Дедупликация абстрактов
Задача 2.1: Сборщик OAI-PMH. Фоновый сервис для выкачивания архива препринтов. Критично: Внедрение SemaphoreSlim (1 запрос раз в 3 секунды).
Задача 2.2: XML Парсинг. Безопасный парсинг Atom XML с помощью XDocument, паттерн "Ранний возврат" при получении тега <error> вместо HTTP 400.
Задача 2.3: Хранение и дедупликация (Upsert). Сохранение только аннотаций (Abstracts) и метаданных. При появлении новой версии статьи (v2) — перезапись документа с флагом "требует пересчета эмбеддинга".
Задача 2.4: Индексы RavenDB. Создание MapReduce индексов для мгновенной агрегации (например, расчет Обитаемой зоны на этапе индексации).

Фаза 2: Семантика, "Прогрев" и RAG-конвейер (Спринты 3–4)
Цель: Обогатить сырые аннотации математическим смыслом. Развернуть Ollama, настроить бесперебойную генерацию векторов (Cache Warming) и реализовать гибридный поиск.
Спринт 3: Локальная LLM и "Прогрев" (Warming) базы В этом спринте мы настраиваем ИИ так, чтобы его падения или нехватка памяти не ломали бэкенд.
Задача 3.1: Инфраструктура Ollama.
Развернуть Ollama локально или в контейнере.
Загрузить модель nomic-embed-text (для быстрого создания векторов) и легковесную LLM типа llama3 или mistral (для саммаризации неопределенностей).
Задача 3.2: Механизм RavenDB Data Subscriptions.
Создать подписку на коллекцию Papers, отфильтрованную по признаку HasEmbeddings == false.
Написать C#-воркер, который открывает постоянное TCP-соединение с БД и забирает статьи батчами (например, по 50 штук).
Задача 3.3: Инференс и механизм ACK (Подтверждение).
Воркер отправляет текст абстракта в API Ollama, получает вектор (массив float[]).
Критично: Реализовать паттерн Acknowledgment. Воркер отправляет команду ACK в RavenDB только после успешной записи вектора в документ. Если Ollama упадет с OOM (Out of Memory), RavenDB не пометит батч как прочитанный и безопасно отдаст его после перезапуска.
Задача 3.4: Приоритетный прогрев (Priority Queue).
Настроить правила подписки так, чтобы статьи за последние 6 месяцев или статьи, упоминающие планеты из "Обитаемой зоны", обрабатывались воркером в первую очередь.
Спринт 4: CQRS, Векторный поиск и Гибридные запросы Связываем "прогретую" базу с логикой приложения.
Задача 4.1: Внедрение CQRS (MediatR).
Строго разделить ответственность: Команды (GenerateEmbeddingsCommand) меняют состояние базы, Запросы (SearchExoplanetsQuery) работают только на чтение через индексы.
Задача 4.2: Настройка Vector Search в RavenDB 7.0.
Создать векторные индексы по полям абстрактов.
Протестировать нативную функцию vector.search(), убедиться в правильности расчета косинусного сходства (Cosine Similarity).
Задача 4.3: Движок Гибридных Запросов (Hybrid Query Engine).
Написать эндпоинты, которые склеивают два мира. Пример логики:
Извлечь из RavenDB планеты, подходящие под жесткие фильтры (pl_masse < 2.0).
Взять поисковый запрос пользователя ("влияние пыли") -> превратить в вектор через Ollama.
Выполнить поиск по вектору в рамках отфильтрованных планет.
Задача 4.4: API для "Uncertainty Tracking".
Создать эндпоинт, который забирает из таблицы ps (сырые данные публикаций) конфликтующие измерения по одной планете.
Отправлять этот контекст в Ollama (модель llama3) с системным промптом для генерации аналитической сводки (почему данные разнятся).

Фаза 3: Визуализация и Риал-тайм (Спринты 5–6)
Цель: Создать рабочее место исследователя, которое реагирует на новые открытия в реальном времени.
Спринт 5: Outbox паттерн и WebSockets
Задача 5.1: Паттерн Outbox. При успешной векторизации статьи сохранять PaperProcessedEvent в рамках той же транзакции базы данных.
Задача 5.2: Интеграция SignalR. Фоновый Dispatcher читает Outbox и пушит события в веб-сокеты клиентов.
Задача 5.3: Авто-тегирование (RavenDB AI ETL). Настройка скриптов классификации внутри БД (например, выставление флага "HWO Candidate").
Спринт 6: Фронтенд-дашборд и 3D (Three.js)
Задача 6.1: Каркас UI. React. Панели фильтров, поисковая строка RAG и слушатели SignalR.
Задача 6.2: 3D-движок (Three.js). Процедурная генерация планет. Связь метаданных с визуалом (если RAG нашел "стеллярный джиттер" — звезда пульсирует; если пыль — рендерится кольцо тумана).
Задача 6.3: Панель Аналитики Неопределенностей. UI для вывода сгенерированных LLM объяснений по расхождениям в данных.
Задача 6.4: E2E Тестирование. Проверка полного цикла от ингестии до изменения 3D-модели на клиенте.

Технологии и библиотеки

1. Backend: Ядро и API (C# / .NET)
Мы используем экосистему .NET, так как она идеально подходит для высоконагруженных фоновых задач (BackgroundServices), надежна в многопоточности (что критично при работе с лимитами arXiv) и имеет отличную типизацию для научных данных.
Платформа: .NET 10.0 (ASP.NET Core Web API)
Почему: Актуальная LTS-версия, встроенная поддержка Rate Limiting, улучшенная производительность JSON-сериализации.
Архитектурный паттерн: Clean Architecture + CQRS
Почему: Проект сложный. Нам нужно жестко отделить парсинг XML от arXiv от логики расчета "Обитаемой зоны" и сохранения векторов в БД.
Библиотеки для ядра:
MediatR: Реализация паттерна CQRS (разделение команд на запись и запросов на чтение).
Polly: Обязательная библиотека для отказоустойчивости (Resilience). Политики Retry, Circuit Breaker и Timeout при запросах к NASA и arXiv.
System.Xml.Linq (XDocument): Встроенный инструмент .NET для парсинга ответов от OAI-PMH (arXiv).
System.Text.Json: Для сериализации данных NASA (работает быстрее NewtonSoft, поддерживает кастомные конвертеры для "плохих" чисел).
Фоновые задачи (Jobs):
Quartz.NET или Hangfire: Для управления инкрементальным обновлением (Sync). Позволяют настроить cron-задачи (например, "каждый день в 02:00") и имеют удобный дашборд для мониторинга успешности парсинга.
2. База данных: Хранение и Поиск
Здесь наш выбор предопределен архитектурой, и это огромное преимущество.
СУБД: RavenDB 7.0+
Почему это идеальный выбор:
Документоориентированность: Идеально ложится на структуру статей и параметров планет (без жесткой SQL-схемы, которая ломается при отсутствии данных).
Vector Search: Нативная поддержка векторного поиска (косинусное сходство) прямо в БД. Нам не нужно разворачивать отдельные векторные БД типа Pinecone, Milvus или Qdrant. Вектор лежит в том же JSON-документе, что и статья.
Data Subscriptions: Надежный механизм для передачи батчей статей в LLM на векторизацию. Не теряет прогресс при падении воркера.
MapReduce Indexes: Позволяет агрегировать астрономические параметры асинхронно, не нагружая процессор при пользовательских запросах.
Клиент БД: Официальный C# клиент RavenDB.Client.
3. Искусственный Интеллект: RAG и Семантика
Мы придерживаемся стратегии локального ИИ (Local AI), чтобы избежать гигантских счетов за API OpenAI или Anthropic при векторизации 50 000+ статей.
Платформа Инференса: Ollama
Почему: Простейший способ локально запустить и масштабировать open-source модели. Имеет удобный REST API, который легко интегрировать с C#.
Модели (Models):
Для векторизации (Embeddings): nomic-embed-text (очень быстрая, требует мало VRAM, длина контекста 8192 токена — идеально для абстрактов).
Для аналитики (Uncertainty Tracking): llama3:8b или mistral:7b. Это легковесные, но умные модели, способные сравнить два текста и выдать структурированное резюме (почему данные радиуса планеты в двух статьях отличаются).
Интеграция с C#:
Microsoft.SemanticKernel (опционально) или прямые HTTP-запросы (через HttpClient) к API Ollama. Для начала рекомендую прямые запросы, так как нам нужна максимальная прозрачность при передаче абстрактов.
4. Frontend: Дашборд и Визуализация
Интерфейс должен быть реактивным и уметь работать с 3D-графикой.
Фреймворк: React.
Стилизация и UI-компоненты: Tailwind CSS + готовая библиотека компонентов (например shadcn/ui для React) для быстрого создания научной панели управления (кнопки, слайдеры, таблицы).
Связь с Backend (Риал-тайм): SignalR
Почему: Официальная технология Microsoft для WebSockets. Нужна для пуш-уведомлений о том, что статья успешно векторизована и 3D-сцена должна обновиться.
3D Визуализация (Killer Feature): Three.js
Почему: Индустриальный стандарт для WebGL.
Экосистема: React — React Three Fiber (R3F). Они радикально упрощают работу с 3D, используя компонентный подход.

Резюме стека (The Stack)
C# 10.0 Web API (MediatR, Polly, Quartz.NET)
RavenDB 7.0 (Vector Search, Subscriptions, MapReduce)
Ollama (nomic-embed-text, llama3)
React + Tailwind CSS + SignalR + Three.js
